# 用户操作手册

本手册用于后续修改和迭代信息筛选网站。目标是让维护者知道“改哪里、看哪里、怎么验证”，不需要先理解全部代码。

## 1. 网站主要用途

本网站用于筛选科技、金融、新闻三类信息。

核心流程是：

1. 配置信息来源。
2. 抓取 RSS 或官方 API 数据。
3. 统一数据格式。
4. 过滤低价值内容。
5. 去除重复信息。
6. 给信息评分。
7. 生成网站展示数据。

## 3. 日常查看网站

如果只是查看页面，可以直接打开：

- `index.html`

如果页面没有显示最新内容，优先检查：

1. `src/data/latest-items.json` 是否存在。
2. `public/site-config.json` 是否配置正确。
3. 浏览器控制台是否有加载文件失败的提示。

## 4. 新增或删除信息来源

按信息类型选择对应文件：

- 科技：`config/sources.tech.json`
- 金融：`config/sources.finance.json`
- 新闻：`config/sources.news.json`

新增来源时建议记录：

- 来源名称
- 来源链接
- 来源类型：RSS 或 API
- 所属分类
- 可信度
- 是否启用

删除来源前建议先确认：

1. 是否长期无法访问。
2. 是否重复率过高。
3. 是否经常产出低价值内容。
4. 是否可以先降权，而不是直接删除。

## 5. 调整过滤规则

过滤规则放在：

- `config/filter-rules.json`

适合在这里处理：

- 屏蔽低价值关键词。
- 屏蔽无关主题。
- 限制过旧信息。
- 排除明显广告、营销、转载内容。

调整建议：

1. 先少量修改规则。
2. 观察是否误删重要内容。
3. 确认有效后再扩大规则范围。

## 6. 调整去重规则

去重规则放在：

- `config/dedupe-rules.json`

适合在这里处理：

- 同一个链接重复出现。
- 多个来源转载同一篇内容。
- 标题高度相似的信息重复出现。

如果发现重复信息太多，可以提高去重强度。

如果发现不同信息被错误合并，应降低去重强度。

## 7. 调整评分规则

评分规则放在：

- `config/scoring-rules.json`

评分通常参考：

- 来源可信度
- 发布时间新旧
- 关键词重要性
- 信息分类
- 是否来自官方来源
- 是否与重点关注主题相关

调整建议：

1. 高价值来源可以加权。
2. 噪音来源可以降权。
3. 重要关键词可以加分。
4. 标题党、广告、重复内容应减分。

## 8. 更新展示数据

当前数据处理流程参考：

1. 抓取 RSS。
2. 标准化数据。
3. 过滤数据。
4. 去重数据。
5. 评分数据。
6. 生成 `src/data/latest-items.json`。

如果修改了来源、过滤、去重或评分规则，应重新生成展示数据，然后打开页面检查结果。

## 9. 每次迭代建议检查

每次修改后建议检查：

- 页面是否能正常打开。
- 三个频道是否都有内容。
- 信息标题、来源、时间、评分是否显示正常。
- 是否有明显重复内容。
- 是否误删了重要内容。
- 高分内容是否真的值得优先阅读。

## 10. 来源复查规则

建议每月复查一次来源质量。

可以删除或停用的来源：

- 长期无法访问。
- 长期没有更新。
- 重复率太高。
- 低质量内容太多。
- 与科技、金融、新闻主题关系不大。

可以提高权重的来源：

- 更新稳定。
- 原创内容多。
- 准确度高。
- 对决策有帮助。
- 经常产出高价值信息。

## 11. 迭代记录建议

每次重要修改建议记录：

- 修改日期
- 修改了哪些来源
- 修改了哪些规则
- 为什么修改
- 修改后效果如何

推荐记录到：

- `docs/feedback-rules.md`
- `docs/source-list.md`
- `docs/scoring-strategy.md`

## 12. 最小维护原则

后续迭代时优先遵守：

1. 不一次性加入太多来源。
2. 不一次性大幅调整评分。
3. 先观察真实效果，再扩大改动。
4. 优先提高信息质量，而不是追求信息数量。
5. 删除来源前先确认原因，避免误删有价值来源。
